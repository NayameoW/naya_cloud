## 2.2 涉众分析
### 2.2.1 涉众分析过程
![7A1046594617F85FC2DAB77EF81639B4.png](https://s2.loli.net/2023/02/15/Si75plNYbTxnGhF.jpg)

涉众分析主要任务包括：
1. 涉众识别：寻找软件系统的涉众类别
2. 涉众描述：
- 描述不同涉众类别的简单特征，包括个人特征和工作特征
- 描述不同涉众类别的复杂特征，包括关注点和兴趣取向，重要性和影响力，输赢条件和受影响程度。
3. 涉众评估：
- 为涉众类别划分优先级
- 评估不同涉众类别的风险，化解风险
- 分析涉众冲突，实现共赢
4. 涉众代表选择：从每种涉众类别中选择代表
5. 制定涉众代表参与需求开发乃至软件系统的参与策略

### 2.2.2 涉众识别——ADM 模型
主体依赖模型 ADM（Actor Dependency Model）
- 目标依赖：依赖者希望被依赖者满足一个条件，但不会规定怎样满足该条件
- 软目标依赖：特殊类型的目标依赖，其条件是无法量化描述的
- 任务依赖：依赖者希望被依赖者执行特定任务。任务依赖比目标依赖更具体，因为满足目标可以执行很多任务，被依赖者有自己的选择权。而任务依赖直接为被依赖者规定了任务。
- 资源依赖：依赖者希望被依赖者提供资源实体（抽象信息或者食物材料）为自己所用，但不关注提供资源需要被依赖者的行为和解决的问题。

### 2.2.4 共赢分析——Stakeholder/Issue
1.  列出系统的**所有涉众类别**，明确描述他们的**兴趣**和**对系统的期望**
2.  从涉众们的**兴趣和期望**中发现背后涉及的**共同问题**(Issue)
3.  建立涉众类别和问题的**关联**，如果某个涉众类别对一个 Issue **存在兴趣**，那么该涉众类别和这个 Issue 就存在**关联**关系
4.  对每一个 Stakeholder-Issue 关系，标明该关系上面所被**寄予的期望**

![72747BDD63288ED97CAF4DECFD58ECBD.png](https://s2.loli.net/2023/02/15/aWgfHYJEcCbnIGV.jpg)

**具体例子**
一家制造业公司计划开发一种新型可重复使用的包装材料，以减少其产品在运输过程中所产生的环境污染。该公司需要使用 stakeholder-issue 模型来解决如下问题：
1. 确定利益相关者：利益相关者包括公司、客户、供应商、员工、政府机构、环保组织等。
2. 分析利益相关者：公司需要考虑利益相关者的需求和期望。客户希望得到更环保的产品，供应商希望减少成本，员工希望更好的工作条件和工资待遇，政府机构和环保组织希望减少环境污染。 
3. 识别问题：对于每个利益相关者，公司需要识别他们所面临的问题和障碍。例如，客户可能不愿意支付更高的价格购买可重复使用的包装材料，供应商可能不愿意改变生产流程，员工可能需要接受新的培训和工作规范，政府机构和环保组织可能需要确保新包装材料符合相关的环保标准。
4. 制定解决方案：针对每个问题，公司需要制定相应的解决方案。例如，公司可以提供客户更多的环保选项，提供供应商更好的协调和沟通，为员工提供培训和支持，与政府机构和环保组织进行合作以确保符合环保标准。 
5. 执行方案：公司需要执行方案并监控其实施过程，以确保它们真正实现共赢效果。同时需要调整方案以适应变化。

## 2.3 面谈、原型和观察
### 2.3.1 联系和区别
面谈是通过与利益相关者进行对话和交流，收集关于系统需求的信息。面谈通常是定向和结构化的，旨在确保收集到准确、完整和可行的需求。面谈可以在不同的阶段进行，包括项目开始前、需求分析阶段和开发过程中。面谈的优点是可以获得深入、全面和明确的需求，以及建立良好的人际关系。缺点和局限性在于比较耗时，时间成本高；被会见者地理分散情况下难以实现；参与者的记忆和交流能力对结果影响较大等等。
原型是一种交互式模型，通常用于呈现需求规范和设计概念。原型是系统设计的早期版本，通过模拟系统功能和交互来激发用户反馈。通过原型，用户可以看到系统如何工作，并提供反馈和建议。原型的优点是可以快速、有效地获得用户反馈，并验证需求的可行性和有效性。
观察是通过观察用户在特定环境下使用系统，收集关于需求和用户行为的信息。观察可以帮助识别用户的需求、行为和偏好，并确定系统的需求和约束。观察通常在实际环境下进行，例如用户办公室或工作场所。观察的优点是可以获得真实和直接的用户反馈，了解用户的实际需求和行为。
这三种方式之间的联系在于，它们都是为了**收集和验证系统需求**而设计的。面谈、原型和观察在不同的阶段和环节中发挥不同的作用，可以相互补充。例如，面谈可以在需求分析阶段提供深入的需求细节，原型可以在设计阶段测试和验证设计方案，观察可以在实施和部署阶段检查系统是否符合用户需求。因此，在进行需求分析工程时，可以根据需要和实际情况，灵活应用这三种方式，以获取和验证准确、完整和可行的系统需求。

### 2.3.2 面谈问题的设计
面谈准备的关键：问题类型
准备问题不仅仅是要清楚**面谈的内容**，更要清楚**提问题的方法**
**问题类型**
1. 开放式问题 (Open-Ended / Closed)
2. 封闭式问题
开放式问题和封闭式问题比较：
- 在会见者对事实核问题的掌握比较有限，希望被会见者能够提供丰富信息时，可以使用开放式问题。一般早期阶段较多使用开放式问题。
- 在会见者对事实和问题的范围比较确定，只是希望被会见者进行选择和确认时，可以使用封闭式问题。一般后期阶段较多使用封闭式问题。
3. 程序性提示
4. 探究式问题
5. 诱导式问题
6. 元问题

### 5.1.2 需求验证的方法
1. 需求评审
评审又称为同级评审，是指由作者之外的其他人来检查产品问题，是主要的静态分析手段。

![643861A62AAFA0F6D6B75DCD00908952.png](https://s2.loli.net/2023/02/15/w2HTGPrIjDR6d1S.jpg)

3. 开发测试用例
要求为每个需求都开发测试用例。通常一条需求的满足可能需要很多个测试用例才能完全体现出来。同时，一个测试用例可能会被用来测试多条需求。

4. 用户手册编制
用户手册主要包含以下内容：
- 对软件系统功能的实现和描述。
- 系统没有实现的功能部分。
- 问题和故障的解决。
- 系统的安装和启动。

5. 利用跟踪关系
建立需求之间的跟踪关系：业务需求（系统特性）->用户需求（业务、任务）->系统级需求（分析模型）
上述链条中，每一个前项都可以跟踪到后项，后项是对前向的展开和细化。

## 5.2 需求管理
### 5.2.1 需求基线
需求基线就是被明确和固定下来的需求集合，是项目团队需要在某一特定产品版本中实现的特征和需求集合。

### 5.2.2 需求基线的维护
1. 配置管理
	- 标识配置项
	- 版本控制
	- 变更控制
	- 访问审计
	- 状态报告
2. 状态维护

### 5.2.3 需求跟踪
以软件需求规格说明文档为基线，在向前和向后两个方向上，描述需求以及跟踪需求变化的能力。
![E0A4AC3F1C5E17617202C25AA0847677.png](https://s2.loli.net/2023/02/15/mIPseu2ADTbMlak.jpg)

**两种跟踪方法**
1. 前向跟踪
- 向前跟踪到需求：说明涉众的需要和目标产生了哪些软件需求
- 从需求向后追溯：说明软件需求来源于哪些涉众的需要和目标
2. 后向跟踪
- 从需求向前跟踪：说明软件需求是如何被后续的开发物件支持和实现的。
- 回溯到需求的跟踪：说明各种系统开发的物件是因为什么原因（软件需求）而被开发出来的。

**实现方法**
- 矩阵：信息清晰易懂，但是限于矩阵的二维性，仅能表达二元的跟踪关系
- 实体关系模型：可以表达多元跟踪关系，但是不够直观
- 交叉：主要被用来在文档之间建立跟踪联系，比较直接，利于使用，但是只适用于对文档的处理

**建立流程**
1. 认识到需求跟踪的重要性，明确需求跟踪需要解决的问题          
2. 说明需求跟踪过程的目标
3. 明确需要捕获的跟踪联系
4. 组织提供资源支持和技术支持
5. 制定有效的过程策略：需求跟踪过程与实际的项目开发工作融合，作为项目开发工作的一部分。
6. 便利需求跟踪信息的使用：为客户、项目管理者以及开发者等项目涉众提供便利的使用途径。

### 5.2.4 需求变更控制
需求的变化是正当和不可避免的
- 问题发生了改变
- 环境发生了改变
- 需求基线存在缺陷
- 用户变动
- 用户对软件的认识变化
- 相关产品的出现

**变更控制委员会（CCB）**
评价需求的变更，做出批准或者拒绝变化的决定，并确保已批准变化的实现。
可能由来自下列部门的人员组成：
- *项目或程序管理部门*
- *产品管理或者需求分析部门*
- *开发部门*
- *测试或者质量保障部门*
- 市场或客户代表
- 编写用户文档的部门
- 技术支持或帮助部门
- *配置管理部门*

**注意事项**
1. 认识到变更的必要性，并为之制定计划
2. 维护需求基线，审计变更记录
3. 管理范围蔓延
4. 灵活应对变更请求
5. 使用辅助工具
